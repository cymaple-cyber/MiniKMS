## 摘要（300–500 字，最后撰写）

**写作结构：**

1. **背景：** 密钥管理是密码系统中最易出错的环节；当 Host 不可信时，Master Key 不应与业务 API 同处一进程。
2. **方法：** 提出 Host + Enclave 分离架构；采用信封加密；实现远程认证与 Enclave 内 DEK 解包及 AES-GCM 加解密。
3. **实现：** 基于 Python/FastAPI 与模拟 TEE 完成原型；含 JWT/RBAC/审计日志；8 项自动化测试通过。
4. **结论：** 在受控威胁模型下，可将敏感密钥操作隔离至 Enclave；需明确与真实硬件 TEE 的边界。

**关键词：** 密钥管理系统；信封加密；可信执行环境；远程认证；AES-GCM；应用密码学

## 第 1 章 绪论

### 1.1 研究背景与动机

- 密码算法相对成熟，但密钥的生成、存储、使用、轮换、销毁仍是工程难点。
- 云环境、多租户、不可信 Host 场景下，KMS 若将 Master Key 放在 API 进程，一旦服务被攻破，全部 DEK 面临泄露风险。
- TEE 提供了「小范围可信计算基」的思路，适合承载密钥敏感操作。

### 1.2 问题定义

> 在 Host 不可信的前提下，如何设计并实现一套可运行的 KMS，使 Master Key 与 DEK 的敏感操作不暴露在 Host 内存中？

### 1.3 研究目标

1. 设计两层密钥结构（Master Key → DEK → 业务数据）。
2. 实现密钥全生命周期管理（创建、轮换、禁用、吊销、销毁）。
3. 将敏感密码操作迁移至模拟 Enclave。
4. 建立 Host ↔ Enclave 远程认证与安全通信协议。
5. 提供可复现实验与测试证据。

### 1.4 主要贡献

| 编号 | 贡献 |
|------|------|
| 1 | 轻量级 KMS 架构，含 RBAC、审计、版本化密钥轮换 |
| 2 | Host/Enclave 分离的信封加密实现，DEK 不出 Enclave |
| 3 | 简化版远程认证 + AES-GCM 会话信道 |
| 4 | 威胁模型与系统边界分析 |
| 5 | 可一键启动的原型系统与集成测试 |

## 第 2 章 相关技术与理论基础

### 2.1 对称密码与认证加密（AEAD）

- AES-256-GCM 的机密性与完整性保证。
- Nonce 唯一性、Associated Data 的作用。
- 选择 GCM 而非 CBC 的理由（认证加密、避免 padding oracle 等）。

### 2.2 密钥派生与信封加密

- Master Key / DEK 分层结构。
- 信封加密（Envelope Encryption）概念。
- 用户口令哈希（bcrypt）与系统密钥的区别。

### 2.3 密钥管理系统（KMS）

- KMS 在 PKI、云安全中的角色。
- 密钥生命周期：Create → Use → Rotate → Revoke → Destroy。
- 与 HSM、云 KMS（AWS KMS 等）的对比（突出本项目的教学/原型定位）。

### 2.4 可信执行环境（TEE）

- TEE 基本思想与可信计算基（TCB）。
- Intel SGX、ARM TrustZone 概念简介。
- 远程认证（Remote Attestation）、密封存储（Sealed Storage）。
- **本文采用软件模拟 TEE**，说明与真实硬件的差异。

### 2.5 身份认证与访问控制

- JWT Bearer 认证机制。
- RBAC 三角色模型（Admin / Key Manager / App User）。
- 审计日志在安全系统中的作用。

## 第 3 章 系统威胁模型与安全需求

### 3.1 系统角色与信任边界

```text
[Client] ──HTTPS/JWT──▶ [Host: MiniKMS API + DB + 审计]
                              │
                    远程认证 + 加密信道
                              ▼
                        [Enclave: Master Key + 加解密]
```

**信任假设：**

- Enclave 进程相对可信（模拟）。
- Host、DB、被动网络窃听者不可信。
- 合法 API 用户通过 JWT 鉴权。

### 3.2 防护目标

| 威胁 | 缓解措施 |
|------|----------|
| 数据库泄露 | 仅存包装后的 DEK，无 Master Key |
| Host 内存窥探 | DEK 解包在 Enclave 内完成 |
| 冒充 Enclave | 远程认证（measurement + 挑战-响应） |
| 信道篡改/重放 | AES-GCM 会话加密 + 序列号 |
| 未授权 API 调用 | JWT + RBAC |

### 3.3 明确不防护（边界，必须写清）

- 物理攻击、侧信道攻击。
- Enclave 进程本身被完全攻破。
- 同时控制 Host 与 Enclave 的攻击者。
- 软件模拟 TEE 不等价于 SGX 硬件隔离。

### 3.4 安全需求清单

- **SR1：** Master Key 不得出现在 Host 环境变量/内存（可信模式）。
- **SR2：** 每次数据加密使用随机 nonce。
- **SR3：** 解密失败返回统一错误，不泄露内部细节。
- **SR4：** 密钥轮换后旧版本仍可解密历史密文。
- **SR5：** 销毁密钥后擦除 DB 中密钥材料。
- **SR6：** 关键操作写入审计日志。

## 第 5 章 关键协议与算法实现

### 5.1 信封加密实现

- DEK 生成：`os.urandom(32)`。
- Master Key 包装 DEK：AES-GCM，AAD = `minikms-dek-v1`。
- 数据加密：AAD = `key_id`。

### 5.2 远程认证协议

**图 5-1：远程认证时序图**

```text
Host ── attestation_request ──▶ Enclave
Host ◀── measurement + identity_key ── Enclave
Host ── challenge(nonce) ──▶ Enclave
Host ◀── HMAC 签名 ── Enclave
Host ── encrypted(session_key) ──▶ Enclave
Host ◀══ 安全信道建立 ══▶ Enclave
```

### 5.3 安全信道

- 会话密钥建立方式。
- 消息格式：长度前缀 + AES-GCM 密文。
- 序列号防重放。

## 核心结论（可置于摘要与第 8 章）

> 本文设计并实现了一套 Host/Enclave 分离的轻量级 KMS。在「Host 不可信、Enclave 相对可信」的威胁模型下，Master Key 仅驻留于 Enclave，DEK 的解包与数据加解密在 Enclave 内完成，Host 与数据库仅保存包装后的密钥材料与业务密文。实验表明，该系统能正确支持密钥生命周期管理与加解密服务，并通过集成测试验证远程认证与 Enclave 后端联通。该方案属于应用密码学工程原型，为后续对接真实 TEE 与后量子密钥封装提供了可扩展基础。
